# C# 비효율/문제 코드 패턴

---

## 문제

`아래 3가지 메서드에서 각각 잠재적인 문제를 파악하라.`

```csharp
// 예제 1
static string BuildReport(List<string> items)
{
    string result = "";
    foreach (string item in items)
    {
        result += item + ", ";
    }
    return result;
}

// 예제 2
static int ParseUserInput(string input)
{
    try
    {
        return int.Parse(input);
    }
    catch (Exception)
    {
        return -1;
    }
}

// 예제 3
static string ReadConfig(string path)
{
    StreamReader reader = new StreamReader(path);
    string content = reader.ReadToEnd();
    return content;
}
```

**Q1.** `items`의 크기가 수천~수만 개로 커질 경우 `BuildReport`에서 발생할 수 있는 성능 문제는 무엇인가요? 어떻게 개선할 수 있나요?

**Q2.** `ParseUserInput`은 파싱 실패 시 `-1`을 반환합니다. 어떤 잠재적인 문제가 있으며, 더 안전하게 작성하려면 어떻게 바꿔야 할까요?

**Q3.** `ReadConfig`는 파일을 정상적으로 읽었음에도 장기적으로 시스템에 악영향을 줄 수 있습니다. 무엇이 문제인가요?

## Q1. 루프 내 문자열 누적 — 성능 문제

**질문:** `items`의 크기가 수천~수만 개로 커질 경우 `BuildReport`에서 발생할 수 있는 성능 문제는 무엇인가요? 어떻게 개선할 수 있나요?

**내 답변**

C# 문자열은 참조 타입이며 불변(immutable)이다. `+=` 연산을 할 때마다 새로운 힙 객체가 생성되고 이전 객체는 GC 대상이 된다. 이는 GC 발생 및 성능 이슈를 유발한다.

> **평가** ✅ 핵심 정확. 불변성과 힙 할당, GC 압박까지 연결한 설명이 올바름.

**이상적인 답변**

`string`은 불변이라 `+=` 할 때마다 새 힙 객체를 생성하고, 이전 문자열의 **각 char를 하나씩** 새 힙 객체에 복사한다.

복사 단위는 문자열 객체가 아니라 **개별 char 1개**이므로, 누적 복사 횟수가 n²이 된다.

예) `["A", "B", "C", "D"]` 누적 시:

| 반복 | 새 힙에 쓰는 char 수 |
|------|---------------------|
| `"" + "A"` → `"A"` | 1 (`A`) |
| `"A" + "B"` → `"AB"` | 2 (`A`, `B`) |
| `"AB" + "C"` → `"ABC"` | 3 (`A`, `B`, `C`) |
| `"ABC" + "D"` → `"ABCD"` | 4 (`A`, `B`, `C`, `D`) |

n개면 총 `1 + 2 + ... + n = n(n+1)/2` → **O(n²)**

개선: `StringBuilder`를 사용하면 내부 버퍼에 char를 누적하므로 힙 할당이 최소화된다.

```csharp
static string BuildReport(List<string> items)
{
    var sb = new StringBuilder();
    foreach (string item in items)
    {
        sb.Append(item);
        sb.Append(", ");
    }
    return sb.ToString();
}
```

## Q2. 예외 삼키기 — catch (Exception)과 -1 반환

**질문:** `ParseUserInput`은 파싱 실패 시 `-1`을 반환합니다. 어떤 잠재적인 문제가 있으며, 더 안전하게 작성하려면 어떻게 바꿔야 할까요?

**내 답변**

숫자가 아닌 값이 들어오면 예외가 발생하므로, `int.Parse()` 전에 숫자인지 확인하는 로직이 필요할 것 같다.

> **평가** ⚠️ 방향은 맞으나 핵심 2가지를 놓침. C#의 관용적인 해결법(`TryParse`)과 `-1` 반환의 모호성 문제를 언급하지 못함.

**이상적인 답변**

문제점 3가지:

1. **`-1`이 유효한 정수와 구분 불가** — `"-1"` 입력 시 파싱 성공이지만 실패로 오인할 수 있다. 실패를 표현하려면 `int?` (nullable)을 반환해야 한다.
2. **예외를 제어 흐름으로 사용** — `int.Parse()`는 실패 시 예외를 던지는데, 예외는 비용이 크다. `int.TryParse()`는 변환 성공 시 `true`, **실패 시 `false`를 반환**하며 예외를 던지지 않는다.
3. **`catch (Exception)`이 너무 광범위** — `OutOfMemoryException`, `StackOverflowException` 같은 복구 불가능한 예외까지 삼켜버린다. 이런 예외를 잡아서 `-1`로 무시하면 프로그램이 비정상 상태로 계속 실행되어, 나중에 전혀 다른 곳에서 이상한 버그가 발생하고 원인 추적이 매우 어려워진다.

```csharp
static int? ParseUserInput(string input)
{
    if (int.TryParse(input, out int value))
        return value;
    return null;
}
```

## Q3. IDisposable 미해제 — 리소스 누수

**질문:** `ReadConfig`는 파일을 정상적으로 읽었음에도 장기적으로 시스템에 악영향을 줄 수 있습니다. 무엇이 문제인가요?

**내 답변**

모르겠다.

> **평가** ❌

**이상적인 답변**

`new StreamReader(path)` 호출 시점에 **파일을 열고 OS 파일 핸들을 획득**한다.
`StreamReader`는 `IDisposable`을 구현하며, 이 파일 핸들은 `Dispose()`를 호출해야 반환된다.

`Dispose()`를 호출하지 않으면:
- 파일 핸들이 GC 수거 전까지 **계속 열려 있음**
- 중간에 예외 발생 시 핸들이 **영원히 닫히지 않을 수 있음**
- 같은 파일을 다른 곳에서 쓰려 하면 **잠금 충돌** 발생 가능

`using` 선언을 쓰면 스코프를 벗어날 때 — 예외가 나도, return해도 — `Dispose()`가 **반드시** 호출된다.

```csharp
static string ReadConfig(string path)
{
    using StreamReader reader = new StreamReader(path);
    return reader.ReadToEnd();
}
```
