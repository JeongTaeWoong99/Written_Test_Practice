# 클래스 설계와 런타임 예외

---

## 문제

`아래 두 클래스에서 각각 잠재적인 문제를 파악하라.`

```csharp
public class Employee
{
    public string Name;
    public double HourlyRate;
    public double HoursWorked;
}

public class PayrollCalculator
{
    private List<Employee> _employees = new List<Employee>();

    public void AddEmployee(Employee employee)
    {
        _employees.Add(employee);
    }

    public double GetTotalPayroll()
    {
        double total = 0;

        for (int i = 0; i <= _employees.Count; i++)
        {
            total += _employees[i].HourlyRate * _employees[i].HoursWorked;
        }

        return total;
    }

    public Employee GetHighestPaid()
    {
        Employee highest = null;

        foreach (Employee employee in _employees)
        {
            if (employee.HourlyRate * employee.HoursWorked > highest.HourlyRate * highest.HoursWorked)
            {
                highest = employee;
            }
        }

        return highest;
    }
}
```

**Q1.** `GetTotalPayroll()`과 `GetHighestPaid()` 두 메서드 모두 실행하면 런타임 예외가 발생합니다. 각각 어떤 예외이고, 원인은 무엇인가요?

**Q2.** `Employee` 클래스의 설계에서 객체지향 원칙 관점으로 어떤 문제가 있나요?

---

## Q1. 두 메서드의 런타임 예외

**질문:** `GetTotalPayroll()`과 `GetHighestPaid()` 두 메서드 모두 실행하면 런타임 예외가 발생합니다. 각각 어떤 예외이고, 원인은 무엇인가요?

**내 답변**

- `GetTotalPayroll()` — `<=` 조건 때문에 범위를 벗어난 인덱스에 접근해 예외가 발생한다.
- `GetHighestPaid()` — `highest`가 `null`인 상태에서 필드에 접근하면 예외가 발생한다.

> **평가** ✅ 두 케이스 모두 원인과 예외 발생 지점 정확.

**이상적인 답변**

**`GetTotalPayroll()` → `IndexOutOfRangeException`**

`i <= _employees.Count` 조건으로 인해 마지막 반복에서 `i == Count`가 되고,
`_employees[Count]`는 존재하지 않는 인덱스이므로 예외가 발생한다.

```csharp
// 수정
for (int i = 0; i < _employees.Count; i++)
```

**`GetHighestPaid()` → `NullReferenceException`**

`highest = null`로 초기화된 상태에서 첫 반복에 `highest.HourlyRate`에 접근하면 예외가 발생한다.

```csharp
// 수정 — 빈 리스트 먼저 처리, 그 다음 첫 요소로 초기화
if (_employees.Count == 0) return null;
Employee highest = _employees[0];
```

> `_employees[0]` 만 쓰면 리스트가 비어있을 때 `ArgumentOutOfRangeException`이 또 발생하므로, 빈 리스트 가드가 반드시 함께 있어야 한다.

## Q2. Employee 클래스의 설계 문제

**질문:** `Employee` 클래스의 설계에서 객체지향 원칙 관점으로 어떤 문제가 있나요?

**내 답변**

`Employee`가 class(참조 타입)라 List 순회 시 메모리 점프가 발생한다. struct로 작성하는 게 바람직하다.

> **평가** ⚠️ 성능 관점의 답변으로, 방향이 다름. 질문이 요구한 건 객체지향 원칙 관점이며, 핵심은 캡슐화 위반이다.

**이상적인 답변**

**캡슐화(Encapsulation) 위반** — `public` 필드를 직접 노출하고 있다.

외부에서 아무 제약 없이 값을 변경할 수 있어, 잘못된 상태가 되어도 막을 방법이 없다.

```csharp
employee.HourlyRate = -999; // 검증 불가
employee.Name       = null; // 검증 불가
```

프로퍼티와 생성자를 사용하면 setter를 통해 유효성 검사를 강제할 수 있고,
내부 구현이 바뀌어도 외부 코드에 영향이 없다.

```csharp
public class Employee
{
    public string Name        { get; private set; }
    public double HourlyRate  { get; private set; }
    public double HoursWorked { get; private set; }

    public Employee(string name, double hourlyRate, double hoursWorked)
    {
        if (hourlyRate < 0) throw new ArgumentException("시급은 0 이상이어야 합니다.");
        Name        = name;
        HourlyRate  = hourlyRate;
        HoursWorked = hoursWorked;
    }
}
```

struct/참조 타입 메모리 관점은 성능 최적화 주제로, OOP 설계 질문과는 다른 관점이다.
